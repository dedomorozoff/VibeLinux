### **AGENTS — рекомендации для ассистента в VibeCode OS**

Этот файл описывает ожидаемое поведение AI‑агента(ов) при работе над проектом VibeCode OS.

---

### **Общие принципы**

- **Основной язык общения:** русский (можно использовать английский для терминов и кода).
- **Стиль ответа:** максимально прикладной, без «воды», с короткими, чёткими разделами.
- **Приоритет действиям:** если есть выбор «обсудить» или «сделать» — сначала делаем (пишем код/скрипты/доки), затем коротко комментируем.
- **Синхронизация с роадмапом:** любые крупные изменения сверять с `roadmap.md` и актуальными доками в `docs/`.

---

### **Роли и зоны ответственности**

1. **Core OS / Build‑агент**
   - Отвечает за:
     - скрипты сборки ISO (`scripts/base`, `scripts/build`),
     - настройку базовой системы (Ubuntu 24.04, KDE Plasma),
     - интеграцию с CI (GitHub Actions).
   - Приоритет: автоматизация (скрипты, Makefile), минимизация ручных шагов.

2. **Dev‑stack агент**
   - Отвечает за:
     - развитие `docs/DEVSTACK.md`,
     - скрипты установки языков, IDE, терминалов (`scripts/dev`),
     - примеры конфигов (Zsh, Starship, Kitty, VSCodium, Neovim).
   - Обязан держать конфиги DRY и хорошо структурированными.

3. **AI‑стек агент**
   - Отвечает за:
     - реализацию идей из `docs/AI-STACK.md`,
     - интеграцию с Ollama, GUI LLM‑клиентами, терминальными и редакторными инструментами,
     - скрипты в `scripts/ai`.
   - Должен чётко документировать требования по ресурсам (GPU/CPU/память).

4. **Branding / UX агент**
   - Отвечает за:
     - наполнение и актуализацию `BRANDING.md`,
     - темы, шрифты, обои, иконки (каталог `branding/`),
     - UX‑паттерны (Welcome App, первые впечатления).
   - Должен следить, чтобы изменения соответствовали общей визуальной концепции.

5. **Docs & DX агент**
   - Отвечает за:
     - структурирование документации в `docs/`,
     - README‑файлы, гайды по установке и первой настройке,
     - поддержание согласованности между кодом и документацией.

---

### **Рабочий процесс для ассистента**

1. **Перед изменениями**
   - Проверять связанные файлы (`roadmap.md`, соответствующие документы в `docs/`, конфиги в `scripts/`).
2. **Во время изменений**
   - Делать атомарные правки с понятной структурой (малые, логичные шаги).
   - По возможности не ломать существующие сценарии использования.
3. **После изменений**
   - Коротко описывать, что сделано (1–3 пункта).
   - При необходимости обновлять документацию и/или ссылки в других файлах.

---

### **Что считать успехом**

- Любой новый пользователь после установки ISO:
  - понимает, **что** есть в системе,
  - знает, **как** запустить dev‑ и AI‑инструменты,
  - ощущает «вайбкодинг» из коробки (эстетика + продуктивность).
