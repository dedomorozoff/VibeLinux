### Сборка alpha-ISO VibeCode OS (черновик)

Этот документ фиксирует выбор стека сборки ISO и базовый сценарий запуска, соответствующий `roadmap.md` (Фаза 1) и текущей структуре `scripts/`.

---

### 1. Выбор стека сборки

- **Базовый подход:** полностью скриптуемый CLI-стек на основе `debootstrap`, без использования GUI-инструментов (Cubic и т.п.).
- **Инструменты:**
  - `debootstrap` — создание минимальной Ubuntu 24.04 (noble) в chroot.
  - `squashfs-tools` (`mksquashfs`) — упаковка rootfs в SquashFS.
  - `xorriso` — подготовка образа.
  - `grub-mkrescue` (пакеты `grub-pc-bin`, `grub-efi-amd64-bin`) — создание загрузочного ISO.
- **Причины выбора:**
  - Хорошо подходит для автоматизации и CI (GitHub Actions).
  - Не требует графического окружения на машине сборки.
  - Легко версионируется и развивается через bash-скрипты в `scripts/`.

---

### 2. Точка входа сборки

- Основной скрипт сборки: `scripts/build/build-iso.sh`.
- Управление режимом работы — через переменную окружения `BUILD_MODE`:
  - `dry-run` (по умолчанию) — проверка зависимостей и структуры, без реальной сборки ISO.
  - `full` — зарезервировано под полноценную сборку alpha-ISO (будет реализовано после фиксации пакетных списков и слоёв Core OS).

Примеры запуска:

```bash
# Локальная проверка стека и окружения
BUILD_MODE=dry-run ./scripts/build/build-iso.sh

# В будущем, при готовности полного пайплайна
sudo BUILD_MODE=full ./scripts/build/build-iso.sh
```

---

### 3. Концептуальный пайплайн сборки (full режим)

В текущей реализации `full` ещё не выполняет реальные шаги, но целевая цепочка выглядит так:

1. **Bootstrap базовой системы:**
   - `debootstrap --arch=amd64 noble "$CHROOT_DIR" "$UBUNTU_MIRROR"`.
   - Настройка `sources.list`, базовой локали, hostname.
2. **Установка Core OS и MATE в chroot:**
   - Установка базовых CLI-утилит (см. `scripts/base/base-packages.sh` и итоговый список пакетов для alpha).
   - Установка MATE (minimal/standard) и необходимого набора приложений.
   - Применение cleanup-скриптов (`scripts/base/cleanup.sh`) и базового брендинга.
3. **Подготовка SquashFS и структуры ISO (live-режим):**
   - `mksquashfs "$CHROOT_DIR" "$IMAGE_DIR"/live/filesystem.squashfs`.
   - Копирование ядра, initrd и конфигурации загрузчика в структуру вида:
     - `$IMAGE_DIR/boot/...`
     - `$IMAGE_DIR/live/filesystem.squashfs`
     - служебные файлы для live-сессии (аналогично стандартной разметке Ubuntu LiveCD).
4. **Создание ISO:**
   - `grub-mkrescue -o "$ISO_OUTPUT" "$IMAGE_DIR"`.

На этапе alpha CI будет выполнять только dry-run, а полноценная сборка (включая live-окружение) может запускаться вручную на более мощной машине.

---

### 4. Live-режим vs установленная система

- **Live-сессия по умолчанию:**
  - ISO грузится в live-окружение MATE с тем же rootfs, который затем используется установщиком.
  - На рабочем столе (или в меню) доступен ярлык установщика Ubuntu/VibeCode OS.
- **Установленная система:**
  - После установки пользователь получает практически тот же набор пакетов, что и в live-режиме, за вычетом специфических live-утилит.
  - Дополнительная настройка (например, установка проприетарных NVIDIA-драйверов) выполняется post-install скриптами (`scripts/drivers/install-nvidia.sh` и др.).
- **Персистентность:**
  - Для alpha-образа персистентные live-сессии не являются приоритетом.
  - При необходимости поддержка persistence может быть добавлена отдельным шагом (через разметку носителя и параметры загрузчика), не влияя на базовую схему сборки.

---

### 4. Интеграция с GitHub Actions

- Workflow: `.github/workflows/build-iso.yml`.
- Текущая логика:
  - Установка зависимостей: `debootstrap`, `squashfs-tools`, `xorriso`, `grub-pc-bin`, `grub-efi-amd64-bin`.
  - Запуск `scripts/build/build-iso.sh` в режиме `dry-run`, чтобы:
    - гарантировать наличие всех CLI-инструментов;
    - проверить, что скрипт сборки не падает и готов к дальнейшему развитию.

По мере развития пайплайна full-сборка может быть вынесена в отдельный job/workflow, чтобы не ломать быстрые smoke-тесты.
